## 役割情報の扱い

画像仕様は、利用者、統括担当、実行担当、監査担当を役割として定義している。
現在の実装では、監査担当はこのプラグインが追加する。一方で、統括担当と
実行担当は独自に新設せず、Hermes が持つ委任構造と役割情報を第一根拠として
扱う。

| 画像上の役割 | Hermes での根拠 | 現在の扱い |
|---|---|---|
| 利用者 | Hermes に依頼を送る外部利用者 | プラグインでは定義しない。 |
| 統括担当 | 利用者直下の親作業単位、または子を委任する側の作業単位 | `delegate_task` を呼ぶ側、親の `session_id`、親子関係から判断する。 |
| 実行担当 | `delegate_task` で作られる子作業単位 | `child_session_id` と `child_role` を記録し、監査材料にする。 |
| 監査担当 | `advisor_audit` と `advisor_resolution_gate` | プラグインが追加する監査用の道具と記録処理として実装する。 |

Hermes の `role` は、主に委任によって作られる子に付く情報として扱う。
そのため、子が末端作業担当か、さらに委任できる調整担当かは、Hermes が渡す
役割情報を正とする。

| Hermes の役割情報 | 意味 | 画像仕様での読み替え |
|---|---|---|
| `leaf` | 末端の子作業単位。さらに委任しない。 | 実行担当 |
| `orchestrator` | さらに子へ委任できる子作業単位。 | 親から見ると実行担当、自分の子に対しては統括担当 |

したがって、設計上の原則は次のとおり。

1. Hermes が役割情報を渡している場合は、それを第一根拠にする。
2. 役割情報が足りない場合は、親子関係、作業単位の識別子、道具呼び出しから補助的に判断する。
3. 独自の統括担当や実行担当の体系は作らない。
4. 監査担当だけをプラグインで追加する。

## 画像仕様には含まれない補足

| 項目 | 画像仕様上の扱い | 理由 |
|---|---|---|
| 監査担当が実行担当を直接操作できないこと | 未達ではない | 画像では、監査担当は実行せず、統括担当が実行と最終判断を担う。 |
| Hermes 本体を改造していないこと | 未達ではない | 画像は本体改造を必須としていない。現在は公式拡張機能で実現している。 |

## 客観的な判定

現在の状態は、画像仕様に対して「中核の監査基盤は実現済み、全体運用仕様は
一部未達」である。

| 領域 | 判定 |
|---|---|
| 役割分担 | 一部達成 |
| 四段階の監査概念 | 達成 |
| 道具登録と実機導入 | 達成 |
| 最終監査の基本動作 | 達成 |
| 計画から最終までの通し保証 | 一部達成 |
| 入力仕様の厳格性 | 一部達成 |
| 記録仕様の完全一致 | 一部達成 |
| 予算、回数、時間制御 | 未達 |
| 画像仕様全体の完全適合 | 未達 |

したがって、現時点で「画像仕様を完全に満たした」と表現してはいけない。

正しい表現は次のとおり。

> Hermes Advisor の中核である、監査担当の分離、四段階監査、実行前の停止、
> 例外と最終回答の監査、指摘処理、記録保存は公式拡張機能で実装し、Pi 上で
> 読み込みと基本動作を確認した。一方で、画像仕様が求める入力形式の厳格化、
> Hermes の役割情報を使った監査判断の強化、予算と時間の制御、会話上の
> 指摘処理保証、全体通し検証は未達である。
