# 画像仕様未達項目の解消方針

> 現行方針: 初期ワークフローは Kanban-only とする。旧来の `delegate_task`
> 前提で書かれた「委任」は、現在は Kanban の task assignment として扱う。
> 最新の実行計画は `docs/exec-plan.md` を参照する。

## 目的

この文書は、`docs/image-spec-compliance.md` の適合表をもとに、未達または
一部達成の項目をどの層で解消するべきか仕分けする。

優先する順序は次のとおり。

1. プラグインだけで解消する。
2. プラグインと呼び出し側の手順書で解消する。
3. 呼び出し側の手順書だけで解消する。
4. 本体改修が理想だが、公式の拡張口、記録、設定、手順で回避する。
5. 直接の本体改修や実行時の差し替えは、公式手段でどうしても満たせない場合の最後の検討対象にする。

ここでいう「呼び出し側の手順書」は、`runtime-profile/skills/advisor-flow/SKILL.md` のように、
統括担当に監査道具の使い方、入力形、指摘処理、最終回答前確認を明示するものを指す。

## 仕分けの分類

| 分類 | 意味 | 基本方針 |
|---|---|---|
| プラグインのみ | 監査道具、割り込み処理、記録、入力検証、設定で解消できる。 | 最優先で実装する。 |
| プラグインと手順書 | 機械的な強制と、統括担当が渡す入力の質の両方が必要。 | プラグインで不足入力を止め、手順書で正しい入力を促す。 |
| 手順書のみ | 行動様式や説明責任の問題で、機械的な完全強制より手順化が向く。 | 手順書を明確化し、最終監査で逸脱を検出する。 |
| 本体改修が理想、回避策あり | Hermes 本体に口があれば自然だが、既存の公式口で近い効果を出せる。 | 直接改修を避け、公式の割り込み処理、記録、最終差し戻しで代替する。 |
| 本体改修なしでは完全不可 | 公式拡張口では完全な保証ができない。 | まず制限を明記し、必要なら上流提案または正式な本体変更として扱う。 |

## 優先順位付き解消表

| 優先度 | 未達または弱い項目 | 解消する層 | 対応方針 | 本体改修回避策 | 直接本体改修の必要性 | 理由 |
|---|---|---|---|---|---|---|
| 1 | 計画監査と委任監査の入力検証が弱い | プラグインと手順書 | 計画監査、委任監査の必須項目を検証する。手順書に入力ひな形を明記する。 | 監査道具の入力検証で不足を拒否し、手順書で正しい入力を作らせる。 | 不要 | 監査道具の引数と検証処理で解消できる。 |
| 2 | Hermes の役割情報を監査判断に十分使えていない | プラグインと手順書 | `child_role`、親作業単位、子作業単位を委任監査と最終監査の判断材料にする。 | 子の開始終了記録と最終監査入力を結び付ける。手順書で役割情報を監査入力に含めさせる。 | 不要 | Hermes は子の役割情報を渡しているため、記録と監査入力で扱える。 |
| 3 | 実Hermes会話での通し検証継続 | プラグインと手順書 | 固定手順とPluginレベルの通し検証は追加済み。次は実会話で、計画、委任、例外、最終、指摘処理、記録保存を継続確認する。 | 手順書に検証手順を置き、プラグインの記録で証跡を確認する。 | 不要 | 実行手順と証跡確認の問題であり、本体改修は不要。 |
| 4 | 指摘処理が会話上に必ず残る保証が弱い | プラグインと手順書 | 指摘処理記録だけでなく、最終回答に処理結果を含めるよう最終監査で確認する。 | 最終監査の入力と最終案を照合し、処理済み指摘の説明不足を修正要求にする。 | 不要 | 最終監査で検出できる。完全な文章生成前強制は不要。 |
| 5 | 最大呼び出し回数、段階別時間制限、予算超過時の縮小処理が弱い | プラグインのみ、一部は手順書 | 監査呼び出し回数、入力量、段階別の制限、縮小時の劣化記録を設定に追加する。 | プラグイン側で回数と入力量を数え、超過時は劣化扱いの監査結果を保存する。 | 不要 | 監査プラグイン自身の予算ならプラグインで管理できる。Hermes 全体予算は対象外として明記する。 |
| 6 | 劣化時の再試行、縮小判断が弱い | プラグインのみ | 監査不能、入力過大、応答不正時の処理規則を実装し、再試行回数と縮小後の記録を残す。 | 劣化結果を通常の監査結果として保存し、最終監査で隠せないようにする。 | 不要 | 監査道具の失敗処理で解消できる。 |
| 7 | 最終改善案が一般論になる可能性がある | プラグインと手順書 | 最終改善案を一件に制限し、実際の証跡に基づく内容だけを許す。 | 監査結果の検証で空または一件に絞り、手順書で採用判断を統括担当に残す。 | 不要 | 出力検証と監査指示で扱える。 |
| 8 | 記録形式が画像の最小データ仕様と完全一致していない | プラグインのみ | 記録項目を画像の最小データ仕様へ寄せる。既存記録との対応表も持つ。 | 別ファイル記録を維持しつつ、画像仕様の項目を追加する。 | 不要 | 記録はプラグイン側が管理している。 |
| 9 | Hermes 標準履歴の上位情報として保存されていない | 本体改修が理想、回避策あり | 標準履歴統合は理想だが、当面は別記録と要約出力で代替する。 | 記録ファイル、監査報告、最終回答内の要約で追跡可能にする。 | 完全一致には必要 | 公式拡張口だけでは標準履歴の内部構造へ自然に保存できない。 |
| 10 | すべての最終回答を生成前に完全停止できない | 本体改修なしでは完全不可 | 既存の確認前割り込みと最終文差し替えで可能な範囲を守る。 | `pre_verify` と `transform_llm_output` を使い、手順書で最終監査を必須化する。 | 完全な生成前停止には必要 | 公式の最終回答前の完全停止口が確認できていない。 |
| 11 | 利用者依頼と統括担当計画のずれを常に自動検出できない | プラグインと手順書 | 計画監査入力に利用者依頼、解釈、作業計画、範囲表、危険度を必須化する。 | 入力がなければ監査道具が拒否し、手順書で必ず含める。 | 不要 | 監査に必要な材料を渡せば検出可能性は上がる。完全自動収集は不要。 |
| 12 | 必要な再確認を必ず実行担当に依頼させる強制力が弱い | 手順書とプラグイン | 監査指摘に「実行担当による再確認が必要」と出た場合、解決記録に証拠を求める。 | 指摘処理記録で証拠不足なら最終監査を通さない。 | 不要 | 監査担当が直接実行しない仕様を保ったまま、証拠要求で補える。 |

## プラグインだけで解消できる項目

| 項目 | 実装内容 | 優先度 |
|---|---|---|
| 計画監査の入力検証 | 利用者依頼、統括担当の解釈、作業計画、範囲表、危険度を必須にする。 | 高 |
| 委任監査の入力検証 | 統括担当の計画、実行担当の範囲、期待する証拠、空振り時の扱い、危険度を必須にする。 | 高 |
| 役割情報の保存と利用 | `child_role`、親作業単位、子作業単位を監査入力と記録に結び付ける。 | 高 |
| 監査回数制限 | 作業単位ごとの監査呼び出し回数を記録し、上限超過時に劣化として扱う。 | 中 |
| 入力量制限の強化 | 現在の最大入力文字数に加え、段階別の上限を持たせる。 | 中 |
| 劣化時の統一処理 | 監査不能、応答不正、入力過大、回数超過を共通の劣化結果として保存する。 | 中 |
| 最終改善案の検証 | 最終改善案を空または一件に制限し、複数提案や一般論を修正要求にする。 | 低 |
| 記録項目の拡張 | 画像の最小データ仕様に合わせた項目を記録に追加する。 | 低 |

## 呼び出し側の手順書で解消できる項目

| 項目 | 手順書で明示すること | 優先度または状態 |
|---|---|---|
| 計画監査の入力品質 | 計画監査には利用者依頼、解釈、作業計画、範囲表、危険度を必ず入れる。 | 高 |
| 委任監査の入力品質 | 委任監査には実行担当の役割、範囲、期待証拠、独立確認の要否を必ず入れる。 | 高 |
| 指摘処理の説明 | 指摘ごとに受理、解決、延期、却下を記録し、最終回答にも要約する。 | 高 |
| 実行担当への再確認 | 監査が再確認を求めた場合、統括担当が必要な実行担当へ依頼し、結果を証拠に入れる。 | 中 |
| 通し検証の手順 | 計画、委任、例外、最終、指摘処理、記録確認を一つの検証手順として実行する。 | 実装済み |
| 最終改善案の扱い | 採用するかどうかは統括担当が判断し、勝手に実施済みにしない。 | 低 |

## プラグインと手順書を組み合わせる項目

| 項目 | プラグイン側 | 手順書側 |
|---|---|---|
| 計画監査の厳格化 | 必須項目不足を拒否する。 | 必須項目の書き方を示す。 |
| 委任監査の厳格化 | 子の役割、範囲、証拠不足を検出する。 | 委任前に役割と範囲を明示させる。 |
| 指摘処理の保証 | 解決記録がない場合は最終前に差し戻す。 | 最終回答に処理状況を説明させる。 |
| 役割情報の利用 | `child_role` と親子関係を記録し、監査の照合対象にする。 | 統括担当に役割情報を監査入力へ含めさせる。 |
| 通し検証 | 記録と監査結果を保存する。 | 検証手順を固定し、期待結果を明記する。 |

## 本体改修を避ける回避策

| 課題 | 本体改修をした場合の理想 | 回避策 | 推奨 |
|---|---|---|---|
| 標準履歴への保存 | Hermes の会話履歴に監査記録を上位情報として保存する。 | 別記録に保存し、最終回答や監査報告に要約を出す。 | 回避策を採用 |
| 完全な最終回答前停止 | 最終回答生成前に必ず監査を通す本体側の割り込み口を追加する。 | 既存の確認前割り込みと最終文差し替えで止める。 | 回避策を採用 |
| 全情報の自動収集 | Hermes 内部状態から依頼、計画、実行担当結果、証拠を自動収集する。 | 手順書で入力を義務化し、プラグインで不足を拒否する。 | 回避策を採用 |
| 役割の完全な型付け | すべての作業単位に統括担当、実行担当、監査担当の型を持たせる。 | Hermes の既存の親子関係と子の役割情報を使う。 | 回避策を採用 |

## 実行時差し替えについて

実行時差し替えは、直接本体ファイルを編集せずに挙動を変える方法ではあるが、
現在の方針では第一候補にしない。

| 観点 | 判断 |
|---|---|
| 公式準拠 | 弱い。公式の拡張口ではなく、内部実装に依存する。 |
| 保守性 | 弱い。本体更新で壊れやすい。 |
| 監査性 | 弱い。どこで挙動が変わったか追いにくい。 |
| 今回の必要性 | 低い。主要未達はプラグインと手順書でかなり解消できる。 |

したがって、実行時差し替えは次の場合だけ検討する。

1. 画像仕様の必須要求である。
2. 公式の拡張口では実現できない。
3. 別記録、手順書、最終差し戻しでは代替できない。
4. 一時的な検証用途であり、正式採用前に本体側の公式対応または上流提案へ移す。

現時点で、実行時差し替えを採用すべき項目はない。

## 推奨する実装順

| 順位 | 作業 | 層 | 理由 |
|---|---|---|---|
| 1 | 指摘処理が最終回答に残るよう、最終監査で確認する。 | プラグインと手順書 | 利用者向け説明不足を防ぐ。 |
| 2 | 監査回数、段階別上限、劣化時処理を追加する。 | プラグイン | 長い運用での安定性を上げる。 |
| 3 | 記録項目を画像の最小データ仕様へ寄せる。 | プラグイン | 追跡性を上げるが、先行項目より緊急度は低い。 |
| 4 | 実Hermes会話で Commander / Worker / Advisor live smoke を継続実行する。 | 手順書と運用 | 実会話上の揺れを確認する。 |

## 実装済みになった項目

| 項目 | 実装状況 |
|---|---|
| 計画監査と委任監査の入力検証 | 計画監査と委任監査の必須項目をプラグインで検証する。 |
| 呼び出し側の手順書の入力ひな形 | `runtime-profile/skills/advisor-flow/SKILL.md` に計画監査、委任監査、最終監査の入力ひな形を追加した。 |
| `child_role` と親子関係の反映 | 子作業単位の開始終了記録を要約し、委任監査と最終監査の監査入力および記録へ反映する。 |
| 通し検証手順 | `runtime-profile/runbooks/live-smoke.md` と `tests/test_end_to_end_flow.py` で、Commander / Worker / Advisor の固定検証手順を追加した。 |

## 結論

未達項目の多くは、Hermes 本体を直接修正せずに、プラグインと呼び出し側の
手順書で解消できる。

本体改修が理想になるのは、標準履歴への完全統合と、すべての最終回答を生成前に
止める制御である。ただし、現時点ではどちらも公式の既存拡張口と運用手順で
実用上の回避ができる。

そのため、次の実装は本体改修ではなく、プラグインの入力検証強化と呼び出し側の
手順書整備から始めるべきである。
